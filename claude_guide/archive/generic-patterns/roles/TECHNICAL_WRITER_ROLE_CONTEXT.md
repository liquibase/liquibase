# Technical Writer Role Context

## Content Standards (v1.0 - Created 2025-01-26)
1. **Only include what we've actually done** - No theoretical knowledge
2. **Evidence required** - Every claim needs proof from our work
3. **Confidence from attempts** - Not self-assessment
4. **Update after retrospectives** - This is a living document
5. **Keep it concise** - If it's not actionable, remove it

*Standards Review: During retrospectives, ask "Are these content standards working?"*

---

## Role Definition
**Primary Responsibility**: Documentation quality, knowledge capture, pattern libraries, developer experience

## Validated Learnings

### Document Organization Evolution:
- **Learning**: Content standards at top prevent degradation
- **Evidence**: Documents without standards became theoretical
- **Application**: All documents now have standards section

- **Learning**: Goal-based organization beats role-based
- **Evidence**: User confusion with role-only structure
- **Application**: Created goal documents in role folders

- **Learning**: State machine concept captures process
- **Evidence**: Clearer understanding after visualization
- **Application**: Created executable state machine system

## Proven Patterns

### Content Standards Pattern
- **What Works**: Standards at top of each document type
- **Why It Works**: Immediate reminder of quality criteria
- **When to Use**: Every document creation
- **Success Rate**: Prevented theoretical content creep

### Living Documentation
- **What Works**: Update immediately after learnings
- **Why It Works**: Fresh memory, accurate capture
- **When to Use**: After every retrospective
- **Success Rate**: High accuracy in captured learnings

## Anti-Patterns (What Failed)

### Documentation After Implementation
- **What We Tried**: Document patterns after coding
- **Why It Failed**: Missing key decisions and context
- **What to Do Instead**: Document during implementation
- **Failure Rate**: Lost important sequence learnings
